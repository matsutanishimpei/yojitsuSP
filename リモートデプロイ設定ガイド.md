# Yojitsu SP リモートデプロイ設定ガイド

この手順はCloudflare Workers、Pages、D1とGitHub Actionsを使う本番配備用です。mainへpushしただけではデプロイされません。`SP_DEPLOY_ENABLED=true` のときだけ配備します。

## 1. 事前条件

- `npm run check` が成功している
- app D1 `yojitsu-db` を削除・変更しないことを確認している
- SP専用D1 `yojitsu-sp-db` を作成できる空きがある
- Cloudflare Pagesの本番URLとWorkersの本番URLが決まっている
- [実運用テストチェックリスト](doc/production-test-checklist.md)の担当者が決まっている

## 2. SP専用D1を作成する

```bash
npx wrangler d1 create yojitsu-sp-db
```

返されたIDを `packages/backend/wrangler.toml` の `DB.database_id` に設定します。`SOURCE_DB.database_id` はappのIDのまま変更しません。

```bash
npx wrangler d1 migrations apply yojitsu-sp-db --remote --config packages/backend/wrangler.toml
npx wrangler d1 time-travel info yojitsu-sp-db --config packages/backend/wrangler.toml
```

`yojitsu-db` にmigrationを適用してはいけません。

## 3. Workerの秘密情報・設定

次をCloudflare Workerへ設定します。

| 名前 | 種別 | 内容 |
|---|---|---|
| `JWT_SECRET` | Secret | 十分に長いランダムなJWT署名鍵 |
| `GAS_SECRET_TOKEN` | Secret | GASとの共有トークン |
| `GBIZINFO_API_KEY` | Secret | 使用する場合のみ |
| `SYNC_ALERT_WEBHOOK` | Secret/変数 | 同期失敗通知先。任意 |
| `ADMIN_ID` | 変数 | 移行互換用の管理ID |
| `ALLOWED_ORIGIN` | 変数 | Pagesの本番オリジン。末尾スラッシュなし |
| `GAS_EMAIL_URL` | 変数 | GAS Webhook URL |

少なくとも `JWT_SECRET` はSecretとして設定し、リポジトリへ書きません。

## 4. GitHub Actionsの設定

GitHubの Settings → Secrets and variables → Actions で設定します。

### Secrets

| 名前 | 内容 |
|---|---|
| `CLOUDFLARE_API_TOKEN` | Workers・Pages・D1に必要な最小権限のAPIトークン |
| `CLOUDFLARE_ACCOUNT_ID` | CloudflareアカウントID |
| `VITE_API_URL` | Workersの本番URL。例 `https://yojitsu-sp-backend.example.workers.dev` |

### Variables

| 名前 | 初期値 | 内容 |
|---|---|---|
| `SP_DEPLOY_ENABLED` | `false` または未設定 | 本番配備ゲート |
| `VITE_API_URL` | 任意 | Secretを使わない場合のAPI URL |

APIトークンは、対象アカウント・対象プロジェクトに限定した最小権限にします。

## 5. 初回配備と確認

1. `SP_DEPLOY_ENABLED` を未設定のままmainへpushし、checkだけ成功することを確認します。
2. DB ID、migration、Worker設定、GitHub設定を別の担当者と照合します。
3. `SP_DEPLOY_ENABLED=true` に変更します。
4. Actionsから配備成功を確認します。
5. 教員ログイン後、手動同期を1回実行します。
6. 同期履歴、appとの件数、SP独自編集の保持を確認します。
7. 実運用テストチェックリストを実施します。

フロントエンドビルドでは `VITE_API_URL` からCSPの `connect-src` を生成します。本番CIはURL未設定なら失敗するため、無制限な通信許可にはなりません。

## 6. ロールバック

- 問題があれば最初に `SP_DEPLOY_ENABLED=false` にして自動配備を止めます。
- DB変更前に `d1 time-travel info yojitsu-sp-db` で復旧可能時点を記録します。
- 復旧対象がSP DBであることを二人で確認してからTime Travelを実行します。
- app DBは復旧・migration・手動修正の対象にしません。
- 復旧後は手動同期し、SPオーバーライドと表示結果を再確認します。

詳細は[DB移行・運用手順](doc/sp-database-migration.md)を参照してください。

## 7. 配備を有効にしてはいけない条件

- `wrangler.toml` のSP DB IDが `00000000-0000-0000-0000-000000000000`
- `VITE_API_URL`、`JWT_SECRET`、`ALLOWED_ORIGIN` が未設定
- migrationまたは初回同期が未確認
- `doc/dump.sql` 等の個人情報を含むファイルが公開可能な状態にある
- `npm run check` または実運用テストが失敗している
